# Atlas Testing Guide

A practical, layered testing system that validates Atlas across all three
surfaces (Go runtime, Bubble Tea TUI, Preact web UI) and gates major builds
with a single command.

## Philosophy

- **Reliability over coverage theatre.** Every test in the gate must be
  deterministic. Flaky tests get fixed or removed.
- **Tests run on the same machine devs use.** No hidden CI-only steps.
- **One command to validate a release.** `make verify-release` is the gate.
- **The scorecard never lies.** It is generated by the script, not hand-maintained.

## What each layer covers

| Layer | Lives in | Validates |
| --- | --- | --- |
| **Runtime unit** | `atlas-runtime/internal/**/*_test.go` | Pure logic вҖ” config, agent loop, validate gate, storage queries, mind reflection, skill registry, forge codegen. |
| **Runtime handler** | `internal/modules/*/module_test.go`, `internal/domain/*` | HTTP route shape, auth middleware, CORS, session handling. |
| **Runtime integration** | `internal/integration/runtime_baseline_test.go`, `architecture_guardrails_test.go` | Boots the full server, exercises baseline routes, asserts package import boundaries. |
| **Wiring** | `cmd/atlas-runtime/wiring_test.go` | All services compose without nil panics on startup. |
| **TUI** | `atlas-tui/{config,client,ui}/*_test.go` | Config load/save round-trip, client HTTP contract via `httptest`, root model `Init`/`Update`/`View` smoke (splash + chat construction, window resize, keypress). |
| **Web build gate** | `npm run build` + `tsc --noEmit` | TypeScript correctness and Vite production build succeed. |
| **Cross-surface gate** | `scripts/verify-release.sh` | Aggregates everything and emits the scorecard. |

## Test tiers

| Tier | Target | What it runs | Use when |
| --- | --- | --- | --- |
| **fast** | `make test-fast` | `go vet`, `go build`, `go test -short` (runtime + TUI), `tsc --noEmit` (if web deps installed) | Local feedback loop while coding. ~30s. |
| **standard** | `make test-standard` | Above + full `go test` (no `-short`) + `vite build` | CI on every PR / before push. |
| **release** | `make verify-release` | Above + `go test -race -short` for the runtime + scorecard regeneration | Before tagging a release or `make install` to your daily driver. |

## How to run

```bash
# Local quick check
make test-fast

# Full pre-push validation
make test-standard

# Full release gate (writes docs/testing/atlas-test-scorecard.md)
make verify-release

# Or call the script directly with a tier
./scripts/verify-release.sh fast
./scripts/verify-release.sh standard
./scripts/verify-release.sh release
```

The script always exits non-zero if any required step fails, and always
overwrites `docs/testing/atlas-test-scorecard.md` with the latest run.

## Adding new tests

### Go (runtime or TUI)

Add `<file>_test.go` next to the code under test. Use `httptest` for HTTP
handlers вҖ” never bind real ports. For things that touch `~/Library/Application
Support/ProjectAtlas/`, swap `HOME` with `t.Setenv("HOME", t.TempDir())` so
runs are hermetic. The TUI follows the same convention (see
`atlas-tui/config/config_test.go`).

For Bubble Tea models, the smoke contract is:

1. Send `tea.WindowSizeMsg` first вҖ” most views render empty until sized.
2. Call `Update` with realistic `tea.KeyMsg` / domain messages.
3. Assert `View()` returns non-empty content and the typed model is intact.

See `atlas-tui/ui/app_test.go` for the canonical example.

### Web UI

There is currently no component test runner. Adding one is the next step
(Vitest + `@testing-library/preact` is the recommended path). Until then,
keep components small and lean on the build/typecheck gate.

When you add Vitest:

1. `npm i -D vitest @testing-library/preact happy-dom`
2. Add `"test": "vitest run"` to `atlas-web/package.json`.
3. Add a `web: vitest run` step to `scripts/verify-release.sh` so it joins
   the gate automatically.

## Known limitations

- **No web component tests.** Tracked above. The build gate is the only thing
  catching web regressions today.
- **No headless browser E2E.** Playwright against the runtime + dist would
  catch the cross-surface seams the integration tests can't.
- **TUI tests are smoke-level.** Full chat-stream replays via a fake SSE
  source are a future improvement.
- **No perf benchmarks in the gate.** Add `go test -bench` for the agent loop
  and validate gate before claiming SLA-grade readiness.

## Future improvements

- Vitest for `atlas-web` with shadow snapshots of the chat composer.
- Playwright E2E that drives the dashboard against a live runtime.
- `go test -bench` regression budget for `internal/agent` and `internal/validate`.
- Coverage report uploaded to `docs/testing/coverage.html` as part of the
  release tier.
